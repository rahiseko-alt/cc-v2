# チェックイン / チェックアウト（簡易引継ぎ）

セッションをまたぐ**揮発的な文脈**（ハマりどころ・一時的判断・環境の癖など）を
`docs/handoff.md` に1件だけ持ち回るための仕組み。ロードマップ（`docs/roadmap.html`）に
触れるセッションで発動する。

## 大原則（handoff とロードマップの分離）

- **handoff に書くのは、ロードマップに載らない事項だけ**。作業状態・進捗・criteria/verify/evidence は
  ロードマップの正であり、ここには書かない（二重管理禁止 / AGENTS.md 準拠）。
- **永続化は git が担う**：この環境はセッションごとにコンテナが破棄され、リポジトリは毎回まっさらに
  clone され直す。`docs/handoff.md` を **commit & push しない限り次セッションは必ず空を読む**。
  だからチェックアウトは「上書き＋commit＋push」までで1セット。

## ① チェックイン（セッション開始時 = 読む）

ロードマップの `meta.active` / `meta.next` を報告する儀式と同じタイミングで実行する。

1. `docs/handoff.md` を読む。
2. 前回の特筆事項があれば、それを1〜2行で要約してユーザーに提示する（空なら「引継ぎなし」と伝える）。
3. その内容と `meta.next` を踏まえ、**`AskUserQuestion` で「今回のセッションで何をやるか」を選択肢で質問**する。
   - 選択肢は `meta.next`（次の一手）を第一候補に、handoff が示す残課題があればそれも並べる。

## ② チェックアウト（ロードマップJSONを更新する瞬間 = 書く）

作業ノードを閉じてロードマップ JSON を編集するのは、元々セッション終盤の意図的な操作。
**その同じ瞬間に handoff も更新する**（別途「終了時に思い出す」必要をなくす）。

1. **前回記入事項を消してから**、今回の特筆事項を書く（上書き＝常に最新1件のみ）。
2. 本文の制約：
   - **3行以内**の自由文。
   - **ロードマップ関係の事項は書かない**（進捗・状態・criteria/verify/evidence は禁止）。
   - 書くのは次セッションが知って得する揮発的文脈だけ（例：ハマった箇所、暫定回避、環境の癖）。
   - 特筆事項が無ければ本文は空（見出しのみ）にする。
3. `docs/handoff.md` を保存し、**commit & push** する（これをしないと引継ぎは消える）。

## docs/handoff.md の書式

```
# handoff（次セッションへの特筆事項 / 3行以内・ロードマップ事項は書かない）

<ここに最新1件を上書き。無ければ空>
```
