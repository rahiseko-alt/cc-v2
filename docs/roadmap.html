概要

このファイル1個 ＝ ロードマップ（図）＋その仕様（README）の自己完結ファイル。 リポジトリにまずこれ1個あれば、ここから開始できる。用途は計画先行の進捗管理で、 アプリ開発でも任意の長期計画でも使える汎用の表。

核となる仕組み（状態と作業）

上流＝状態（満たしたい結果）。下流＝作業（実行する操作）。作業を進めると状態が満たされる。

  • 状態ノード（state）：達成したい結果・条件。criteria（受入条件）を持つ。
  • 作業ノード（work）：具体的な操作。status＋commit/done_at を持つ。
  • 層数もタイプ名も固定しない。状態の中に状態でも作業でもぶら下げられる（再帰）。

完了ゲート（品質担保の心臓）

状態は「子が全部 done」だけでは done にしない。受入条件を検証し、証拠が付いて初めて done。

  • 作業ノード：実行して記録（commit/done_at）が残れば done。
  • 状態ノード：子が全部 done → 一旦 検証待ち（黄バッジ ⛔検証待ち）。受入条件が全て検証されて初めて done（赤字）。
  • これが作業→検証→状態成立の一方通行を作る。ここが無いと「子を全部やった＝一応OK」というタグだけ完了になり品質が抜ける。

受入条件（criteria）の書き方

条件は「動くこと」で書く。「ドキュメントがある／レビューした」で書かない。

  • 各条件は text（満たすべき状態）＋ verify（どう確かめるか＝実行可能な手順）＋ evidence（結果の証拠：テスト出力・commit・スクショ）。
  • evidence が空＝未検証（☐）。埋まって初めて充足（☑）。図の 受入 n/m バッジで進捗が見える。
  • 証拠は「◯◯レビュー.md が存在する」を認めない。実物が動いた記録を指す。

分解の規則（不変・自己相似）

最上位はゴール（最終到達状態）。以下、同じ規則を層の深さによらず当て続ける。

  • 不変の規則：P が成立する ⟺ 条件 C1…Cn が全て成立（各々が必要・合わせて十分）。これをどの層でも同じに当てる。層数は設計せず、当て続けた結果として出る。
  • 層は思考の分解粒度であって処理（工程）ではない。setup→deploy のような時系列は背骨に置かない（それは葉を辿る順番＝道順）。組織軸は「何が真であるべきか」。
  • 条件＝子ノード。まだ証明が要る条件は子の状態（自分の criteria を持つ）、直接確かめられる条件は葉の受入条件。再帰の底は必ず作業（操作）。
  • 個数は導出、粒度は選択：子の数は必要十分性から出る（設計しない）。どこまで割るか＝粒度は最初に1回決める計画の契約。

粒度の線：各葉の状態に、実行可能な verify が1つ書けるところまで割る。書けたら止める。その下（操作）は着手時に展開。

  • 浅すぎ＝目的が曖昧で検証不能。深すぎ＝過剰分解で麻痺。線は「意味の分解（何/なぜ）から行動の分解（どう）へ切り替わる高さ」に置く。

割り方（計画先行・道順は都度）

目的地（状態）は先に全部描く。道順（作業）は着手時に割る。コミット1個サイズで止める。

  • 上流の状態は着手前に描き切る。下流の作業は、今の meta.active に着いてから分割・追記（遅延展開）。
  • 掘るときは「◯番を割って」→ その状態の受入条件を満たす作業へ、この規則で分割。
  • 作業1個＝1つの目的＝コミット単位。数は固定しない（2でも6でも）。

状態表示（文字色とバッジ）

  • 黒字 ＝ 未・着手中・検証待ち（todo / doing）
  • 赤字 ＝ 済（done）＝ 作業完了かつ受入条件を全て検証済み
  • 受入 m/m 全条件充足 ／ 受入 n/m 未充足あり ／ ⛔検証待ち 作業は済だが未検証
  • データ上は todo/doing/done/blocked の4値。親（状態）は子＋受入ゲートから自動導出（手動で done にしない）。

現在地・次の一手

meta.active（現在地ノードID）と meta.next（次の一手）を上部バナーに表示。セッション開始時に最初に読む値。

完了は git に紐付け（二重管理しない）

  • 完了の真実は git。作業完了時は status:"done" ＋ commit（短ハッシュ or PR）＋ done_at。
  • 状態の受入条件の evidence も commit / テスト出力 / スクショ等の実記録を指す。事実と食い違わせない。

データと表示の分離（重要）

ロードマップの正はこの HTML 内の JSON（<script id="roadmap-data">）。日々の更新はこの JSON だけを編集する。 状態・作業の判定と受入ゲートは末尾の描画エンジンが担う（モデルを変える時だけ触る）。

カード表記ルール

番号 [種別] テーマ （意味）   受入 n/m  ⛔検証待ち

  • 種別＝状態/作業。テーマ＝専門用語 / 意味＝非エンジニア用語（かっこ囲み）。
  • 状態ノードは展開すると受入条件（☑/☐＋検証方法＋証拠）を表示。

構造・インタラクション

  • 枠なしテキスト＋ツリー線（├─ └─）。1項目=2段（1段目 番号/種別/テーマ/（意味）/受入バッジ、2段目 詳細/完了日/commit）。
  • 折りたたみは <details>（子あり・受入条件ありはキャレット ▸、純粋な葉は非表示）。現在地は青丸の ✓ バッジ。
  • 階層は文字サイズ＋インデント＋接続線で区別（浅いほど大きい：d0=18px / d1=15 / d2=13 / d3=12）。
  • 図シートには解説を置かない（説明はこの README シートに集約）。
