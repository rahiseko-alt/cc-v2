概要

このファイル1個 ＝ 進捗ロードマップ（図）＋その仕様（README）の自己完結ファイル。 リポジトリにまずこれ1個あれば、ここから開始できる。用途は進捗管理で、 アプリ開発でも任意の計画・プロジェクト管理でも使える汎用の表（今回の学習用は一例）。

運用（回し方）

  1. 開始：図タブで現在地（✓ ＝ meta.active）を見て、このセッションでやることを把握。
  2. 計画：全体を葉まで書いておく。割り方は「目的で割る／コミット1個サイズで止める」。
  3. 実行→記録：終わった葉を done（赤字）にし commit / done_at を記録、meta.active を次へ。
  4. 共有：どの AI ツール（Claude Code / Codex / Cursor / Antigravity）でもこの図を参照・更新。データの正は下記 JSON アイランド。

データと表示の分離（重要）

ロードマップの正はこの HTML 内の JSON（<script id="roadmap-data">）。更新はこの JSON だけを編集する。CSS/描画JSは触らない。

割り方の規則

目的で割る。コミット1個サイズになったら止める。

  • 粒度＝コミット単位（1ノード＝1つの目的）。数は固定しない（2でも6でも）。
  • 掘るときは「◯番を割って」→ その枝だけこの規則で分割・追記。

状態（文字色で表現）

  • 黒字 ＝ 未・着手中（todo / doing）
  • 赤字 ＝ 済（done）
  • データ上は todo/doing/done/blocked の4値を保持。表示は「done＝赤字／それ以外＝黒字」。

親の状態は子から自動導出（手動で done にしない）。全子 done → done、それ以外は未完扱い。

現在地・次の一手

meta.active（現在地ノードID）と meta.next（次の一手）を上部バナーに表示。セッション開始時に最初に読む値。

完了は git に紐付け（二重管理しない）

  • 完了の真実は git。ノード完了時は status:"done" ＋ commit（短ハッシュ or PR）＋ done_at（＝コミット日）を記録。
  • done ノードにはコミットを表示。事実と食い違わせない。

カード表記ルール

番号  テーマ  （意味）
詳細  完了日  commit
  • テーマ＝専門用語 / 意味＝非エンジニア用語（かっこ囲み）。改行は意味の後・詳細の前。

作図の作法（デザイン仕様）

他の図に流用してもぶれないための共通仕様。色は必ずトークン経由、値を直書きしない。

構造・インタラクション

  • 枠なしテキスト＋ツリー線（├─ └─）。1項目=2段（1段目 番号/テーマ/（意味）、2段目 詳細/完了日/commit）。
  • 折りたたみは <details>（子ありはキャレット ▸、葉は非表示）。現在地は青丸の ✓ バッジ。
  • 階層は文字サイズ＋インデント＋接続線で区別（浅いほど大きい：d0=18px / d1=15 / d2=13 / d3=12）。子コンテナは必ず display:block（インラインだと字下げが効かない）。
  • 図シートには解説を置かない（説明はこの README シートに集約）。
